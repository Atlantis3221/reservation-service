# Фича: Деплой и финальная сборка

**Ветка:** `06-deploy`
**Статус:** не начат

## Контекст

**Зачем:** всё должно работать на сервере как единый продукт — можно кидать ссылку кому угодно и сервис просто работает.

**Что есть сейчас:** бэкенд деплоится через rsync + docker-compose на финский сервер, фронтенд — через gh-pages. Нет единого процесса сборки, нет проверки полного флоу.

## User Stories

### US-1 — Полный флоу от регистрации до клиента (Приоритет: P1)

Новый пользователь проходит весь путь: `/start` → регистрация бани → настройка расписания → клиент открывает ссылку и видит слоты.

**Почему этот приоритет:** если полный флоу не работает — сервис не готов к показу.

**Как проверить независимо:** пройти весь путь от `/start` до открытия ссылки клиентом на продакшн сервере.

**Сценарии приёмки:**

1. **Дано** чистый сервер с задеплоенным сервисом, **Когда** новый пользователь пишет `/start` боту, **Тогда** проходит онбординг: название → slug → ссылка
2. **Дано** владелец зарегистрирован, **Когда** он задаёт расписание через бота, **Тогда** слоты появляются на фронтенде
3. **Дано** расписание настроено, **Когда** клиент открывает `domain.ru/slug`, **Тогда** видит слоты и может нажать «Написать в Telegram»

---

### US-2 — Данные переживают рестарт (Приоритет: P1)

После `docker-compose restart` или пересоздания контейнеров все данные на месте.

**Сценарии приёмки:**

1. **Дано** есть зарегистрированные бани и бронирования, **Когда** выполнен `docker-compose restart`, **Тогда** все данные сохранены
2. **Дано** контейнеры пересозданы (`docker-compose down && up`), **Тогда** данные сохранены через volume

---

### US-3 — Изоляция между владельцами (Приоритет: P1)

Два владельца работают параллельно — данные полностью изолированы.

**Сценарии приёмки:**

1. **Дано** два владельца зарегистрированы, **Когда** каждый управляет своим расписанием, **Тогда** они не видят данные друг друга
2. **Дано** клиент открывает `/slug-1`, **Тогда** видит только слоты бани 1

---

### US-4 — Единый деплой (Приоритет: P2)

Один скрипт/процесс собирает фронтенд и деплоит всё (бэкенд + фронтенд + nginx) на сервер.

**Сценарии приёмки:**

1. **Дано** изменения в коде, **Когда** запущен деплой, **Тогда** фронтенд собирается и попадает в volume/папку для nginx, бэкенд обновляется через docker-compose

## Ключевые сущности

- **docker-compose.yml**: бэкенд + nginx + volumes (data, frontend static)
- **deploy pipeline**: сборка фронтенда → деплой бэкенда → рестарт

## Edge Cases

- Что если деплой прервался на середине? — контейнеры должны оставаться в рабочем состоянии (предыдущая версия)
- Что если фронтенд не собрался? — деплой бэкенда не должен ломать текущий фронтенд

## Критерии приёмки

- [ ] `docker-compose.yml` обновлён: бэкенд + nginx + volumes
- [ ] Фронтенд собирается и кладётся в volume/папку для nginx
- [ ] Полный флоу работает на продакшне: `/start` → онбординг → расписание → клиент видит слоты
- [ ] `docker-compose restart` — данные на месте
- [ ] Два владельца работают параллельно — данные изолированы
- [ ] Деплой-процесс документирован
