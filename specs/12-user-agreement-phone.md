# Фича: Пользовательское соглашение и передача контакта владельца

**Ветка:** `12-user-agreement-phone`
**Создана:** 2026-03-14
**Статус:** реализовано

## Контекст

**Зачем:**
1. Юридическое соответствие — владелец заведения при первом использовании бота должен принять пользовательское соглашение (оферту сервиса, согласие на обработку персональных данных).
2. Получение контакта — владелец заведения при регистрации делится своим номером телефона через Telegram, чтобы у администраторов сервиса был его телефон для связи.

**Что есть сейчас:** Владелец отправляет `/start` боту, вводит название и slug заведения — заведение создаётся. Никакого соглашения не показывается. Номер телефона владельца не запрашивается. Клиент на сайте только просматривает расписание и переходит в Telegram для бронирования — никаких данных не оставляет.

## User Stories

### US-1 — Пользовательское соглашение при первом использовании бота (Приоритет: P1)

Владелец заведения, впервые взаимодействующий с ботом, должен принять пользовательское соглашение прежде чем сможет зарегистрировать заведение.

**Почему этот приоритет:** Правовое требование — без согласия обработка данных владельца незаконна.

**Как проверить независимо:** Отправить `/start` боту впервые → бот показывает текст соглашения с кнопками → принять → регистрация продолжается. При повторном `/start` соглашение не показывается.

**Сценарии приёмки:**

1. **Дано** новый пользователь (нет заведений в базе), **Когда** он отправляет `/start`, **Тогда** бот отправляет текст пользовательского соглашения и inline-кнопки «Принимаю» / «Отклоняю».
2. **Дано** бот показал соглашение, **Когда** владелец нажимает «Принимаю», **Тогда** согласие сохраняется в базе, и бот переходит к регистрации заведения (запрашивает название).
3. **Дано** бот показал соглашение, **Когда** владелец нажимает «Отклоняю», **Тогда** бот отвечает, что без принятия соглашения использование сервиса невозможно. Регистрация не начинается.
4. **Дано** владелец ранее принял соглашение, **Когда** он отправляет `/start` повторно, **Тогда** соглашение не показывается, бот работает как обычно.

---

### US-2 — Владелец делится контактом при регистрации заведения (Приоритет: P1)

После принятия соглашения и создания заведения бот запрашивает у владельца номер телефона через стандартный механизм Telegram (`request_contact`). Номер сохраняется в базе для внутреннего использования администраторами сервиса.

**Почему этот приоритет:** Администраторам нужен номер телефона владельца для связи.

**Как проверить независимо:** Владелец создаёт заведение → бот просит поделиться контактом → владелец нажимает кнопку → номер сохраняется в базе.

**Сценарии приёмки:**

1. **Дано** владелец завершил ввод названия и slug заведения, **Когда** бот создаёт заведение, **Тогда** бот отправляет сообщение с `ReplyKeyboardMarkup` и кнопкой «📱 Поделиться контактом» (`request_contact: true`).
2. **Дано** бот запросил контакт, **Когда** владелец нажимает кнопку и отправляет контакт, **Тогда** номер телефона сохраняется в поле `owner_phone` таблицы `businesses`, клавиатура убирается, бот подтверждает.
3. **Дано** бот запросил контакт, **Когда** владелец отправляет текстовое сообщение вместо контакта, **Тогда** бот повторно просит нажать кнопку для передачи номера.
4. **Дано** телефон владельца сохранён, **Когда** запрашиваются данные через публичный API `GET /api/business/:slug`, **Тогда** `owner_phone` НЕ возвращается в ответе.

---

## Ключевые сущности

- **AgreementAcceptance**: новая таблица или поле в `businesses` — факт принятия соглашения владельцем (`agreement_accepted_at DATETIME`), привязка к `owner_chat_id`.
- **Business (обновление)**: новое поле `owner_phone TEXT` — номер телефона владельца из Telegram-контакта.

## Edge Cases

- Владелец не поделился контактом сразу → бот напоминает при следующем `/start`, пока телефон не получен.
- Владелец сменил номер → может повторно поделиться контактом через команду `/phone`.
- У владельца несколько заведений → соглашение принимается один раз (привязано к `owner_chat_id`), телефон запрашивается один раз и проставляется всем заведениям этого владельца.

- [x] Без принятия соглашения регистрация заведения невозможна.
- [x] После создания заведения бот запрашивает контакт владельца через кнопку `request_contact`.

- [x] `owner_phone` НЕ отдаётся через публичный API.
