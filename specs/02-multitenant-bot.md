# Фича: Мультитенант — регистрация бани через бота

**Ветка:** `02-multitenant-bot`
**Статус:** реализован

## Контекст

**Зачем:** сейчас бот работает только с одним захардкоженным `ADMIN_CHAT_ID`. Любой человек должен иметь возможность зарегистрировать свою баню через бота и получить персональную страницу.

**Что есть сейчас:** один владелец, один `ADMIN_CHAT_ID` в env, один набор слотов без привязки к бизнесу.

## User Stories

### US-1 — Регистрация бани через бота (Приоритет: P1)

Человек пишет `/start` боту. Бот спрашивает название бани, генерирует slug и отдаёт ссылку `domain.ru/slug`. Баня зарегистрирована, владелец может управлять расписанием.

**Почему этот приоритет:** это ядро мультитенантности — без регистрации невозможно подключить вторую баню.

**Как проверить независимо:** написать `/start` боту с нового аккаунта, пройти онбординг, получить slug, убедиться что страница доступна.

**Сценарии приёмки:**

1. **Дано** новый пользователь, **Когда** он пишет `/start`, **Тогда** бот спрашивает «Как называется ваша баня?»
2. **Дано** пользователь ввёл название «Баня у Петра», **Когда** бот обработал ввод, **Тогда** сгенерирован slug `banya-u-petra` и пользователь получает ссылку
3. **Дано** slug `test` уже занят, **Когда** бот пытается сгенерировать такой же, **Тогда** добавляется суффикс для уникальности

---

### US-2 — Изоляция данных владельцев (Приоритет: P1)

Каждый владелец видит и управляет только своими слотами. Команды `бронь`, `расписание` работают в контексте бани владельца.

**Сценарии приёмки:**

1. **Дано** два владельца зарегистрированы, **Когда** владелец А создаёт бронь, **Тогда** владелец Б не видит её в своём расписании
2. **Дано** незарегистрированный пользователь, **Когда** он пишет команду бронирования, **Тогда** бот предлагает сначала зарегистрироваться

---

### US-3 — Управление настройками (Приоритет: P2)

Владелец может посмотреть и изменить название бани и slug через команду `/settings`.

**Сценарии приёмки:**

1. **Дано** владелец зарегистрирован, **Когда** он пишет `/settings`, **Тогда** бот показывает текущие название и slug
2. **Дано** владелец хочет сменить slug, **Когда** он вводит новый slug, **Тогда** slug меняется и старая ссылка перестаёт работать

## Ключевые сущности

- **businesses**: id (INTEGER PK), slug (TEXT UNIQUE), name (TEXT), owner_chat_id (TEXT), telegram_username (TEXT), created_at (TEXT)
- **slots**: добавляется поле business_id (INTEGER FK → businesses.id)

## Edge Cases

- Что если пользователь уже зарегистрировал баню и снова пишет `/start`? — показать текущую баню, предложить создать ещё одну или изменить настройки
- Валидация slug: только латиница, цифры, дефис; минимум 3 символа
- Что если Telegram username владельца изменился? — обновлять при каждом сообщении
