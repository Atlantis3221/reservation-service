# Фича: Визуальный календарь в админ-панели

**Ветка:** `19-admin-visual-calendar`
**Создана:** 2026-03-16
**Статус:** реализовано (P1), P2 (drag-and-drop, редактирование) — отложено

## Контекст

**Зачем:** Сейчас в админ-панели расписание управляется только через текстовые команды чата (аналог Telegram-бота). Это неудобно для визуального контроля — владелец заведения не видит общую картину расписания «глазами». Визуальный календарь в стиле Google Calendar позволяет быстро понять загрузку, создать запись кликом, перенести бронь драг-н-дропом, отменить бронь — всё в одном интерфейсе.

**Что есть сейчас:**
- В `frontend/src/components/Calendar.tsx` есть read-only календарь (месячный вид + дневной таймлайн) для клиентского фронтенда. Он показывает слоты, но не позволяет ими управлять.
- Админ-панель (`admin/`) — чат-интерфейс. Управление расписанием только через текстовые команды (`/слоты`, `/расписание`, `/запись`, `/отмена`).
- Общей `shared/` папки для переиспользования компонентов между `frontend/` и `admin/` нет.

**Важно:** Mobile-first! Интерфейс должен быть удобен для работы с телефона.

## User Stories

### US-1 — Shared-библиотека календаря (Приоритет: P1)

Календарный компонент выносится в `shared/calendar/` как локальный npm-пакет (workspace), переиспользуемый и в `frontend/`, и в `admin/`. Компонент содержит базовый рендеринг (месячная сетка, дневной таймлайн с навигацией по неделе), а конкретное поведение (клик, создание, перетаскивание) задаётся через props/callbacks.

**Почему этот приоритет:** Без shared-компонента невозможно переиспользовать код — будет дублирование и расхождение.

**Как проверить независимо:** После выноса — `frontend/` продолжает работать как раньше (регрессия), а `admin/` может импортировать и рендерить тот же календарь.

**Сценарии приёмки:**

1. **Дано** календарь в `shared/calendar/`, **Когда** `frontend/` импортирует его, **Тогда** клиентский фронтенд работает без изменений в поведении.
2. **Дано** календарь в `shared/calendar/`, **Когда** `admin/` импортирует его, **Тогда** в админке отображается календарь с месячным видом и дневным таймлайном.

---

### US-2 — Отображение расписания в календаре админки (Приоритет: P1)

Владелец заведения видит в админ-панели визуальный календарь с расписанием: в месячном виде — дни с записями и свободные дни; в дневном виде — таймлайн со всеми слотами (свободные, забронированные, заблокированные), включая имя клиента и телефон для забронированных.

**Почему этот приоритет:** Базовое визуальное отображение — фундамент для интерактивности.

**Как проверить независимо:** Открыть календарь в админке, убедиться что расписание совпадает с тем, что показывает чат-команда `/расписание`.

**Сценарии приёмки:**

1. **Дано** у заведения есть слоты на завтра, **Когда** владелец открывает календарь в админке, **Тогда** он видит завтрашний день отмеченным; при клике видит таймлайн со слотами.
2. **Дано** есть забронированный слот с клиентом «Иванов» и телефоном, **Когда** владелец смотрит дневной таймлайн, **Тогда** он видит блок брони с именем «Иванов», телефоном и временем.
3. **Дано** на день нет расписания (нет available-слотов), **Когда** владелец открывает этот день, **Тогда** он видит пустой таймлайн с кнопкой «Задать расписание».

---

### US-3 — Создание записи из календаря (Приоритет: P1)

Владелец может кликнуть/тапнуть на свободное время в дневном таймлайне — появляется компактная форма (bottom sheet на мобильном) с полями времени начала, конца, имени клиента. Заполнил → сохранил.

**Почему этот приоритет:** Главный use case — быстрое создание записи визуально.

**Как проверить независимо:** Создать запись через календарь, проверить что она появилась и в чат-команде `/расписание`, и в клиентском фронтенде.

**Сценарии приёмки:**

1. **Дано** дневной таймлайн открыт, **Когда** владелец тапает на свободное время и заполняет форму (время, имя клиента), **Тогда** создаётся бронь, блок появляется на таймлайне.
2. **Дано** владелец пытается создать бронь на уже занятое время, **Тогда** показывается предупреждение с возможностью подтвердить или отменить.
3. **Дано** на день нет расписания, **Когда** владелец нажимает «Задать расписание», **Тогда** открывается форма задания рабочих часов на этот день.

---

### US-4 — Отмена записи из календаря (Приоритет: P1)

Владелец может тапнуть на забронированный слот в таймлайне — открывается карточка с деталями и кнопкой «Отменить».

**Сценарии приёмки:**

1. **Дано** есть бронь «Иванов 14:00–15:00», **Когда** владелец тапает на неё и нажимает «Отменить», **Тогда** появляется подтверждение; после подтверждения бронь удаляется, таймлайн обновляется.

---

### US-5 — Редактирование / перенос записи (Приоритет: P2)

Владелец может тапнуть на бронь → «Редактировать» → изменить время или клиента → сохранить. Drag-and-drop на десктопе как дополнительная возможность.

**Почему этот приоритет:** P2, потому что можно обойтись отменой + созданием новой записи; перенос — удобство.

**Сценарии приёмки:**

1. **Дано** бронь «Иванов 14:00–15:00», **Когда** владелец тапает «Редактировать», меняет время на 16:00–17:00, **Тогда** бронь обновляется.
2. **Дано** владелец перетаскивает бронь на занятое время (десктоп), **Тогда** показывается предупреждение с подтверждением.

---

### US-6 — Навигация между чатом и календарём (Приоритет: P1)

В админ-панели появляется нижний таб-бар (mobile-first) с переключением между «Чат» и «Календарь». Оба режима работают параллельно — изменения через чат видны в календаре и наоборот.

**Сценарии приёмки:**

1. **Дано** админ-панель открыта на телефоне, **Когда** владелец тапает на таб «Календарь», **Тогда** он видит визуальный календарь с расписанием текущего заведения.
2. **Дано** владелец создал запись через календарь, **Когда** он переключается на чат и вводит `/расписание`, **Тогда** запись видна.

---

## Ключевые сущности

- **TimeSlot** (существующая): `id`, `businessId`, `dateKey`, `startTime`, `endTime`, `status` (available/booked/blocked), `clientName`, `clientPhone`, `note`
- **SharedCalendar**: переиспользуемый UI-компонент (месячный вид + дневной таймлайн с навигацией по неделе), принимает слоты через props, эмитит события (`onSlotClick`, `onTimeRangeSelect`, `onSlotDrop`)
- **AdminCalendarPage**: страница-обёртка в админке — загрузка данных через Admin API, формы создания/редактирования/отмены, кнопка «Задать расписание»

## Edge Cases

- **Мультипользовательский доступ:** сейчас одно заведение = один админ. В будущем может быть несколько — пока не реализуем защиту от коллизий.
- **Слоты через полночь (22:00–02:00):** отображаются как в текущем фронте — таймлайн расширяется за 24:00.
- **Нет расписания на день:** показываем пустой таймлайн + кнопку «Задать расписание» (вызывает форму задания рабочих часов).
- **Телефон клиента:** показываем в карточке бронирования на таймлайне.
- **Вид:** дневной таймлайн с навигацией по неделе (как на текущем фронте), без отдельного недельного multi-column вида.
