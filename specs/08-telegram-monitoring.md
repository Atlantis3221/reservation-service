# Фича: Мониторинг сервера через Telegram

**Ветка:** `08-telegram-monitoring`
**Создана:** 2026-03-14
**Статус:** черновик

## Контекст

**Зачем:** сервер будет работать на VPS (1 ядро, 1 GB RAM, 20 GB SSD). Нужно видеть состояние сервиса удалённо, не заходя по SSH. Ошибки должны приходить моментально, а ежедневный дайджест — подтверждать, что всё в порядке.

**Что есть сейчас:** эндпоинт `/health` возвращает `{ status: 'ok', timestamp }`. Логи видны только через SSH. Об ошибках узнаём, когда пользователь жалуется.

## User Stories

### US-1 — Автоматические алерты об ошибках (Приоритет: P1)

При uncaught exception, unhandled rejection или ошибке внутри Express/бота — владельцу сервиса приходит сообщение в Telegram с текстом ошибки и стектрейсом.

**Почему этот приоритет:** без этого ты узнаёшь о падении только когда клиент пожалуется.

**Как проверить независимо:** кинуть тестовое исключение, убедиться что сообщение пришло в Telegram.

**Сценарии приёмки:**

1. **Дано** сервер работает, **Когда** происходит uncaught exception, **Тогда** в Telegram приходит сообщение с текстом ошибки и стектрейсом (обрезанным до разумной длины)
2. **Дано** сервер работает, **Когда** происходит unhandled promise rejection, **Тогда** в Telegram приходит сообщение с текстом ошибки
3. **Дано** ошибка в Express-обработчике, **Когда** запрос возвращает 500, **Тогда** в Telegram приходит сообщение с URL и текстом ошибки
4. **Дано** за последние 60 секунд уже отправлен алерт, **Когда** происходит ещё одна ошибка, **Тогда** сообщение НЕ отправляется (rate limiting, чтобы не заспамить)

---

### US-2 — Команда /health в боте (Приоритет: P1)

Владелец пишет `/health` боту и получает текущее состояние сервера: uptime, использование памяти, количество бизнесов и слотов.

**Почему этот приоритет:** возможность в любой момент проверить «жив ли сервер» без SSH.

**Как проверить независимо:** написать `/health` боту, получить ответ с метриками.

**Сценарии приёмки:**

1. **Дано** сервер работает, **Когда** владелец пишет `/health`, **Тогда** бот отвечает с uptime, RAM (rss/heap в MB), количеством бизнесов и слотов в БД
2. **Дано** пользователь НЕ является `MONITOR_CHAT_ID`, **Когда** он пишет `/health`, **Тогда** бот не отвечает (команда игнорируется или отвечает «нет доступа»)

---

### US-3 — Ежедневный дайджест (Приоритет: P2)

Раз в сутки бот отправляет короткий отчёт: uptime, RAM, количество бизнесов, количество бронирований за последние 24 часа.

**Почему этот приоритет:** позволяет не дёргать `/health` руками — каждое утро видишь статус в Telegram.

**Как проверить независимо:** дождаться запланированного времени (или вызвать руками), получить дайджест.

**Сценарии приёмки:**

1. **Дано** сервер запущен, **Когда** наступает заданное время (по умолчанию 09:00 MSK), **Тогда** отправляется дайджест в указанный чат
2. **Дано** сервер только что перезапустился, **Когда** наступает время дайджеста, **Тогда** дайджест содержит корректный uptime с момента перезапуска

## Ключевые сущности

- **MONITOR_CHAT_ID**: chat id для отправки алертов и дайджестов. Отдельная env-переменная (может совпадать с owner_chat_id, но не обязана)
- **Rate limiter**: простой in-memory таймер, не даёт слать больше 1 алерта в минуту
- **Дайджест-таймер**: `setInterval` или `setTimeout` с расчётом до целевого часа

## Edge Cases

- Что если `MONITOR_CHAT_ID` не задан? — мониторинг просто не активируется, сервис работает как раньше
- Что если Telegram API недоступен при отправке алерта? — ловить ошибку, не ронять процесс, логировать в console
- Что если сервер упал до инициализации бота? — uncaught exception handler ставится до `initBot()`, но без бота отправить нельзя; логируем в console
- Стектрейс может быть длинным — обрезать до 1000 символов, чтобы Telegram не отклонил сообщение
- Rate limiting: при шторме ошибок (100 ошибок в секунду) — отправить первую, остальные подавить; после паузы в 60 секунд — снова разрешить

- [ ] Rate limiting: не более 1 алерта в 60 секунд

- [ ] Ежедневный дайджест в заданное время
