# Фича: Подписочная система (рекуррентные платежи)

**Ветка:** `23-subscriptions`
**Создана:** 2026-03-17
**Статус:** черновик

## Контекст

**Зачем:** Сейчас сервис полностью бесплатный. Нужна монетизация — владельцы заведений платят 300 ₽/мес за использование платформы. Подписка должна списываться автоматически раз в месяц, с обработкой всех кейсов (неудачная оплата, просрочка, отмена).

**Что есть сейчас:** Бизнес создаётся бесплатно (через Telegram-бота или админ-панель), без ограничений по времени. Никакой платёжной интеграции нет.

## Выбор платёжного провайдера

**ЮKassa (ex Яндекс.Касса)** — основной кандидат для РФ:

- Поддерживает автоплатежи (recurring) через сохранённые платёжные методы
- Webhooks для уведомлений о статусе платежей
- Хорошая документация, SDK для Node.js
- Комиссия: 3.5% + возможна фиксированная часть (зависит от договора)
- Для подключения нужно: ИП/ООО, договор с ЮKassa, `shopId` + `secretKey`

**Альтернативы:** CloudPayments, Тинькофф Эквайринг. [УТОЧНИТЬ: есть предпочтения по провайдеру?]

## Модель подписки

- **Один тариф:** 300 ₽/мес
- **Подписка привязана к `owner_chat_id`** (владелец), не к отдельному бизнесу — все заведения одного владельца покрываются одной подпиской
- **Биллинговый цикл:** от даты первого успешного платежа (например, оплатил 17 марта → следующее списание 17 апреля)

## User Stories

### US-1 — Первая оплата подписки (Приоритет: P1)

Владелец заведения оформляет подписку и оплачивает первый месяц. Карта сохраняется для последующих автосписаний.

**Почему этот приоритет:** Без первой оплаты вся система не работает.

**Как проверить независимо:** Создать бизнес, перейти на страницу оплаты, оплатить 300 ₽, убедиться что подписка активна.

**Сценарии приёмки:**

1. **Дано** владелец без подписки, **Когда** нажимает «Оформить подписку», **Тогда** открывается платёжная форма ЮKassa на 300 ₽ с сохранением карты
2. **Дано** платёж успешен, **Когда** ЮKassa присылает webhook `payment.succeeded`, **Тогда** подписка активируется, `payment_method_id` сохраняется, `next_payment_at` = текущая дата + 1 месяц
3. **Дано** платёж не прошёл (недостаточно средств, ошибка банка), **Когда** ЮKassa присылает webhook `payment.canceled`, **Тогда** подписка не активируется, владелец видит сообщение об ошибке и может попробовать снова

---

### US-2 — Автоматическое ежемесячное списание (Приоритет: P1)

Система раз в месяц автоматически списывает 300 ₽ с сохранённой карты.

**Почему этот приоритет:** Ядро рекуррентной модели.

**Как проверить независимо:** Дождаться (или эмулировать) наступления даты списания, убедиться что платёж создаётся и обрабатывается.

**Сценарии приёмки:**

1. **Дано** активная подписка с `next_payment_at = сегодня`, **Когда** запускается планировщик (cron), **Тогда** создаётся рекуррентный платёж через ЮKassa API с сохранённым `payment_method_id`
2. **Дано** рекуррентный платёж успешен, **Когда** приходит webhook `payment.succeeded`, **Тогда** `next_payment_at` сдвигается на +1 месяц, `paid_until` обновляется
3. **Дано** рекуррентный платёж не прошёл, **Когда** приходит webhook `payment.canceled`, **Тогда** начинается retry-цикл (см. US-3)

---

### US-3 — Retry при неудачном списании (Приоритет: P1)

Если автоплатёж не прошёл — система повторяет попытки по расписанию, а не блокирует сразу.

**Почему этот приоритет:** Без retry система будет отключать бизнесы при любом временном сбое.

**Как проверить независимо:** Эмулировать неудачный платёж, убедиться что повторы происходят по расписанию.

**Сценарии приёмки:**

1. **Дано** рекуррентный платёж не прошёл (попытка 1), **Когда** прошло 2 дня, **Тогда** система делает попытку 2
2. **Дано** попытка 2 не прошла, **Когда** прошло ещё 3 дня (день 5 от первой попытки), **Тогда** система делает попытку 3
3. **Дано** попытка 3 не прошла, **Когда** прошло ещё 3 дня (день 8 от первой попытки), **Тогда** система делает финальную попытку 4
4. **Дано** все 4 попытки не прошли, **Тогда** подписка переходит в статус `past_due` → `suspended`
5. **Дано** любая retry-попытка прошла успешно, **Тогда** подписка возвращается в `active`, цикл сбрасывается

**Расписание retry:**

| Попытка | Через сколько дней | Описание |
|---------|-------------------|----------|
| 1       | 0 (день списания) | Основная попытка |
| 2       | +2 дня            | Первый retry |
| 3       | +5 дней           | Второй retry |
| 4       | +8 дней           | Финальный retry |

---

### US-4 — Grace period и деактивация (Приоритет: P1)

После исчерпания retry-попыток бизнес не отключается мгновенно — есть grace period.

**Почему этот приоритет:** Нельзя отрезать клиента без предупреждения.

**Сценарии приёмки:**

1. **Дано** подписка в статусе `past_due` (retry идут), **Тогда** бизнес продолжает работать, но владелец видит предупреждение «Оплата не прошла, обновите карту»
2. **Дано** все retry исчерпаны (день 8+), подписка → `suspended`, **Тогда** бизнес деактивируется:
   - Клиентский фронтенд (`/:slug`) показывает «Заведение временно недоступно»
   - Бот перестаёт обрабатывать бронирования для этого бизнеса
   - Данные **не удаляются** — владелец может реактивировать
3. **Дано** подписка `suspended`, **Когда** владелец оплачивает вручную, **Тогда** бизнес снова активен

---

### US-5 — Уведомления о платежах (Приоритет: P2)

Владелец получает уведомления о событиях подписки.

**Почему этот приоритет:** Важно, но не блокирует ядро.

**Сценарии приёмки:**

1. **Дано** первый платёж прошёл, **Тогда** владелец получает сообщение в Telegram-боте: «Подписка оформлена! Следующее списание — [дата]»
2. **Дано** ежемесячное списание прошло, **Тогда** владелец получает: «Списано 300 ₽. Следующее списание — [дата]»
3. **Дано** списание не прошло, **Тогда** владелец получает: «Не удалось списать 300 ₽. Проверьте баланс карты. Следующая попытка — [дата]»
4. **Дано** подписка приостановлена, **Тогда** владелец получает: «Подписка приостановлена. Ваши заведения скрыты для клиентов. Оплатите подписку для восстановления»

---

### US-6 — Отмена подписки владельцем (Приоритет: P2)

Владелец может отменить автопродление.

**Сценарии приёмки:**

1. **Дано** активная подписка, **Когда** владелец нажимает «Отменить подписку», **Тогда** подписка помечается `cancel_at_period_end = true`, автосписание не произойдёт
2. **Дано** подписка с `cancel_at_period_end`, **Тогда** бизнес работает до конца оплаченного периода (`paid_until`), потом деактивируется
3. **Дано** подписка отменена, но период не истёк, **Когда** владелец нажимает «Возобновить подписку», **Тогда** `cancel_at_period_end` снимается, автосписание вернётся

---

### US-7 — Смена карты (Приоритет: P2)

Владелец может привязать другую карту для автосписаний.

**Сценарии приёмки:**

1. **Дано** активная подписка, **Когда** владелец нажимает «Сменить карту», **Тогда** открывается платёжная форма ЮKassa с `save_payment_method: true` на 1 ₽ (холд, не списание)
2. **Дано** новая карта привязана, **Тогда** `payment_method_id` обновляется, следующее списание пойдёт с новой карты

---

### US-8 — Страница управления подпиской в админ-панели (Приоритет: P1)

Владелец видит статус подписки и может управлять ей.

**Сценарии приёмки:**

1. **Дано** нет подписки, **Тогда** владелец видит баннер «Оформите подписку — 300 ₽/мес» с кнопкой оплаты
2. **Дано** подписка активна, **Тогда** владелец видит: статус, дату следующего списания, последние 4 цифры карты, кнопки «Сменить карту» / «Отменить»
3. **Дано** подписка `past_due`, **Тогда** предупреждение красным с кнопкой «Оплатить вручную»
4. **Дано** подписка `suspended`, **Тогда** сообщение об остановке с кнопкой «Оплатить и восстановить»

## Ключевые сущности

### Subscription (подписка)

| Поле                   | Тип     | Описание |
|------------------------|---------|----------|
| `id`                   | INTEGER | PK |
| `owner_chat_id`        | TEXT    | Владелец (FK → owner_agreements) |
| `admin_user_id`        | INTEGER | Админ-пользователь (FK → admin_users), NULL если только через Telegram |
| `status`               | TEXT    | `active` / `past_due` / `suspended` / `canceled` |
| `payment_method_id`    | TEXT    | ID сохранённого метода оплаты в ЮKassa |
| `payment_method_card`  | TEXT    | Последние 4 цифры карты (для отображения) |
| `current_period_start` | TEXT    | Начало текущего периода |
| `current_period_end`   | TEXT    | Конец текущего периода (`paid_until`) |
| `next_payment_at`      | TEXT    | Дата следующего списания |
| `cancel_at_period_end` | INTEGER | 0/1 — отменить после текущего периода |
| `retry_count`          | INTEGER | Текущий счётчик retry (0–4) |
| `created_at`           | TEXT    | Дата создания |
| `updated_at`           | TEXT    | Дата обновления |

### Payment (платёж)

| Поле               | Тип     | Описание |
|--------------------|---------|----------|
| `id`               | INTEGER | PK |
| `subscription_id`  | INTEGER | FK → subscriptions |
| `yokassa_payment_id` | TEXT  | ID платежа в ЮKassa |
| `amount`           | INTEGER | Сумма в копейках (30000 = 300 ₽) |
| `status`           | TEXT    | `pending` / `succeeded` / `canceled` |
| `type`             | TEXT    | `first` / `recurring` / `manual` |
| `error_reason`     | TEXT    | Причина ошибки (если canceled) |
| `created_at`       | TEXT    | Дата создания |
| `updated_at`       | TEXT    | Дата обновления |

### Жизненный цикл подписки

```
[нет подписки]
     │
     ▼ первый платёж
  pending ──(платёж не прошёл)──► [нет подписки]
     │
     ▼ платёж прошёл
  active ◄──────────────────────────────┐
     │                                  │
     ├─(cancel_at_period_end)──► canceled │
     │                     (конец периода)│
     │                                    │
     ▼ рекуррент не прошёл               │
  past_due ──(retry успешен)─────────────┘
     │
     ▼ все retry исчерпаны
  suspended ──(ручная оплата)──► active
```

## Техническая реализация

### Интеграция с ЮKassa

**Первый платёж:**

```
POST https://api.yookassa.ru/v3/payments
{
  "amount": { "value": "300.00", "currency": "RUB" },
  "capture": true,
  "save_payment_method": true,
  "confirmation": {
    "type": "redirect",
    "return_url": "https://admin.slotik.tech/subscription?status=success"
  },
  "description": "Подписка Slotik — первый месяц",
  "metadata": { "owner_chat_id": "123456", "type": "first" }
}
```

**Рекуррентный платёж (серверный, без участия пользователя):**

```
POST https://api.yookassa.ru/v3/payments
{
  "amount": { "value": "300.00", "currency": "RUB" },
  "capture": true,
  "payment_method_id": "<saved_id>",
  "description": "Подписка Slotik — продление",
  "metadata": { "subscription_id": "42", "type": "recurring" }
}
```

### Webhook endpoint

`POST /api/webhooks/yokassa` — принимает уведомления от ЮKassa.

Безопасность:
- Проверка IP-адресов ЮKassa (whitelist)
- Идемпотентность: дедупликация по `yokassa_payment_id`
- HMAC-подпись (если включена в настройках ЮKassa)

### Планировщик (cron)

Запускается раз в час (или через `setInterval`):

1. Выбрать все подписки с `status = active` и `next_payment_at <= now`
2. Для каждой создать рекуррентный платёж
3. Выбрать все подписки с `status = past_due` и наступившим retry
4. Для каждой создать retry-платёж
5. Выбрать все подписки с `cancel_at_period_end = 1` и `current_period_end <= now`
6. Перевести в `canceled`, деактивировать бизнесы

## Edge Cases

1. **Владелец зарегистрировался только через Telegram (нет admin_user)** — как оплачивает? → Бот присылает ссылку на оплату (redirect в браузер), подписка привязывается к `owner_chat_id`
2. **Владелец привязал Telegram к admin-аккаунту** — подписка одна, привязана к `owner_chat_id`, видна и в боте, и в админ-панели
3. **У владельца несколько заведений** — одна подписка покрывает все [УТОЧНИТЬ: или подписка per-business?]
4. **Подписка истекла, но есть будущие бронирования** — данные сохраняются, но фронтенд не показывает бизнес; при реактивации всё восстанавливается
5. **Двойной webhook** — дедупликация по `yokassa_payment_id` (INSERT OR IGNORE)
6. **Владелец платит вручную во время retry-цикла** — retry сбрасывается, подписка возвращается в `active`
7. **Первый платёж: пользователь закрыл страницу ЮKassa** — подписка остаётся в `pending`, платёж истекает через 1 час (по настройкам ЮKassa)
8. **Что показывать для бизнеса без подписки?** — [УТОЧНИТЬ: нужен ли бесплатный период / trial? Например, 14 дней бесплатно после регистрации?]
9. **Существующие бизнесы (уже зарегистрированные)** — [УТОЧНИТЬ: дать им grace period? Например, 30 дней с момента запуска фичи?]
10. **Возвраты (refund)** — [УТОЧНИТЬ: нужна ли возможность возврата? Ручной через ЮKassa-дашборд или автоматический?]

## Что нужно до начала разработки

1. **ИП или ООО** — ЮKassa работает только с юрлицами / ИП
2. **Договор с ЮKassa** — получить `shopId` и `secretKey`
3. **Настроить webhooks** — указать URL `https://slotik.tech/api/webhooks/yokassa` в личном кабинете ЮKassa
4. **Фискализация (54-ФЗ)** — для онлайн-платежей в РФ нужна онлайн-касса. ЮKassa может выступать платёжным агентом и формировать чеки (интеграция с АТОЛ / OrangeData), но это нужно настроить
