# Фича: LLM вместо регулярок для парсинга команд бронирования

**Ветка:** `27-llm-booking-parser`
**Создана:** 2026-03-18
**Статус:** черновик

## Контекст

**Зачем:** Сейчас парсинг команд бронирования работает на жёстких регулярных выражениях. Это создаёт проблемы:
- Пользователь обязан писать в строго определённом формате («сегодня бронь с 14 до 18»)
- Не поддерживаются синонимы: «забронируй», «запиши», «бронирование», «резерв»
- Не поддерживаются конкретные даты: «15 марта», «15.03»
- Не поддерживается «послезавтра», «в следующую пятницу»
- Опечатки и смешение раскладок ломают парсинг
- Каждый новый формат = новая регулярка и новые тесты

LLM решит все эти проблемы, понимая естественный язык без привязки к формату.

**Что есть сейчас:**
- `backend/src/bot/parsers.ts` — 4 парсера на регулярках:
  - `parseBookingCommand` — «в пятницу бронь на 15:00 на 3 часа Иванов»
  - `parseBookingRange` — «сегодня бронь с 14:30 до 18:00 Петров»
  - `parseCancelCommand` — «отмени бронь на сегодня 14:00»
  - `parseFlexibleSchedule` — «на этой неделе ПН-ПТ c 10 до 23»
- `backend/src/bot/handlers.ts` — вызывает парсеры в порядке приоритета, первый совпавший выигрывает
- Нераспознанные команды → `trackUnrecognizedCommand()` + ответ «Не понял команду»

## Принятые решения

- **LLM-провайдер:** GigaChat (нативный доступ из РФ, оплата в ₽)
- **Стратегия:** Regex-first, LLM-fallback — сначала пробуем регулярки (бесплатно, мгновенно), LLM вызывается только если регулярки не распознали
- **Фича-тогглинг:** LLM включается/выключается per business через таблицу `feature_flags`
- **Единый интерфейс:** абстракция `LLMProvider` — в будущем легко добавить DeepSeek, YandexGPT без изменения логики парсинга

## User Stories

### US-1 — Бронирование в свободной форме (Приоритет: P1)

Владелец пишет боту сообщение о бронировании в произвольной форме, и бот корректно его распознаёт и создаёт бронь — если регулярки не справились, а у бизнеса включён фича-флаг LLM.

**Почему этот приоритет:** Основная боль — пользователи не попадают в формат регулярки, и бот не понимает их.

**Как проверить независимо:** Отправить боту сообщения в разных форматах и убедиться, что бронь создаётся.

**Сценарии приёмки:**

1. **Дано** заведение с LLM-флагом ON и расписанием, **Когда** владелец пишет «забронируй пятницу с 14 до 18 на Иванова», **Тогда** регулярка не совпала → LLM парсит → создаётся бронь на пятницу 14:00–18:00, клиент «Иванов»
2. **Дано** заведение с LLM-флагом ON и расписанием, **Когда** владелец пишет «запиши послезавтра с 12 до 15», **Тогда** бронь создаётся на послезавтра
3. **Дано** заведение с LLM-флагом ON и расписанием, **Когда** владелец пишет «сегодня бронь с 14 до 18» (стандартный формат), **Тогда** регулярка распознала → LLM не вызывается → бронь создаётся
4. **Дано** заведение с LLM-флагом ON, **Когда** владелец пишет «бронь на завтра 15-18 Петров +79991234567 день рождения, нужен торт», **Тогда** LLM извлекает clientName=«Петров», clientPhone=«+79991234567», note=«день рождения, нужен торт» → всё сохраняется в слот

---

### US-2 — Отмена брони в свободной форме (Приоритет: P1)

Владелец отменяет бронь в произвольной форме.

**Сценарии приёмки:**

1. **Дано** заведение с LLM ON и существующая бронь, **Когда** владелец пишет «удали бронь на завтра в 14», **Тогда** бронь отменяется
2. **Дано** заведение с LLM ON и существующая бронь, **Когда** владелец пишет «сними бронирование Иванова на пятницу», **Тогда** LLM парсит, бронь отменяется

---

### US-3 — Фича-тогглинг per business (Приоритет: P1)

LLM-парсинг включается/выключается для каждого бизнеса отдельно.

**Сценарии приёмки:**

1. **Дано** заведение с LLM-флагом OFF, **Когда** владелец пишет «забронируй послезавтра с 12 до 15», **Тогда** LLM не вызывается, регулярка не совпала → «Не понял команду»
2. **Дано** заведение с LLM-флагом ON, **Когда** то же сообщение, **Тогда** LLM парсит → бронь создаётся

---

### US-4 — Fallback при недоступности LLM (Приоритет: P1)

Если LLM-сервис недоступен, система продолжает работать на регулярках.

**Сценарии приёмки:**

1. **Дано** LLM-сервис недоступен, **Когда** владелец пишет «сегодня бронь с 14 до 18», **Тогда** регулярка обрабатывает, бронь создаётся
2. **Дано** LLM-сервис недоступен, **Когда** владелец пишет «забронируй послезавтра», **Тогда** пользователь получает «Не понял команду» (а не ошибку сервиса)

---

### US-5 — Расписание и просмотр в свободной форме (Приоритет: P2)

**Сценарии приёмки:**

1. **Дано** заведение с LLM ON, **Когда** владелец пишет «работаем на следующей неделе пн-пт 10-22, сб 12-20», **Тогда** LLM парсит → расписание создаётся
2. **Дано** заведение с LLM ON, **Когда** владелец пишет «что на завтра?», **Тогда** показывается расписание на завтра

## Ключевые сущности

- **LLMProvider** (интерфейс): абстракция над LLM-провайдером. Метод `parseCommand(text: string, currentDate: string): Promise<ParsedCommand>`. Реализации: `GigaChatProvider`, в будущем `DeepSeekProvider` и др.
- **ParsedCommand**: union-тип результата парсинга — `booking | cancel | schedule | show_schedule | unknown`. Для `booking` дополнительные поля: `clientPhone?`, `note?` (описание/пожелание)
- **FeatureFlag**: запись в таблице `feature_flags` — `business_id + flag_name + enabled`. Флаг `llm_parsing`.

## Архитектура

```
Сообщение пользователя
        ↓
   Regex Parsers (sync, бесплатно)
   parseFlexibleSchedule → parseCancelCommand → parseBookingRange → parseBookingCommand
        ↓ (распознано)           ↓ (не распознано)
   Existing handlers      Feature flag «llm_parsing» ON?
                                ↓ (да)           ↓ (нет)
                           LLM Provider      «Не понял команду»
                           (GigaChat API)
                                ↓
                           JSON → Zod validation
                                ↓ (valid)        ↓ (invalid/timeout)
                           Existing handlers  «Не понял команду»
```

```

```

### Ключевые решения

1. **Regex-first, LLM-fallback**: регулярки проверяются первыми (мгновенно, бесплатно), LLM только для нераспознанных
2. **Structured output**: LLM возвращает JSON, валидируемый через Zod
3. **System prompt содержит текущую дату** — для корректного вычисления «завтра», «послезавтра»
4. **GigaChat OAuth**: токен кешируется на 29 мин (живёт 30), обновляется автоматически
5. **Таймаут 5 сек** — если LLM не ответила, считаем как «не распознано»
6. **Единый интерфейс LLMProvider** — добавить нового провайдера = реализовать один метод

## Edge Cases

- LLM вернёт невалидный JSON → Zod отклонит → «Не понял команду»
- LLM «галлюцинирует» дату → валидация: дата не в прошлом, не дальше 60 дней
- Сообщение не про бронирование («привет») → LLM возвращает `intent: "unknown"` → «Не понял команду»
- Два намерения в одном сообщении → обрабатывается только первое
- GigaChat OAuth-токен протух → автообновление перед запросом
- Сертификат Минцифры → отключение проверки TLS для GigaChat endpoint
