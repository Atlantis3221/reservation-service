# Фича: Персистентное хранилище (SQLite)

**Ветка:** `01-sqlite-storage`
**Статус:** не начат

## Контекст

**Зачем:** сейчас данные хранятся в памяти (Map) — рестарт сервера = потеря всех броней. Для продакшна данные должны переживать перезапуск.

**Что есть сейчас:** `schedule.ts` использует `Map<string, SlotStatus>` — все данные теряются при рестарте контейнера.

## User Stories

### US-1 — Данные переживают рестарт (Приоритет: P1)

Владелец бани записал несколько бронирований через бота. Сервер перезапущен (деплой, обновление, сбой). После рестарта все бронирования на месте.

**Почему этот приоритет:** без этого сервис бесполезен в продакшне — одно падение и все данные пропали.

**Как проверить независимо:** записать бронь через бота, сделать `docker-compose restart`, проверить что бронь видна через API и бота.

**Сценарии приёмки:**

1. **Дано** есть бронь «Олег 14:00–18:00 на завтра», **Когда** сервер перезапущен, **Тогда** бронь отображается в расписании через бота
2. **Дано** задано расписание работы бани, **Когда** контейнер пересоздан (`docker-compose down && up`), **Тогда** расписание сохранено

---

### US-2 — Миграция без потери совместимости (Приоритет: P1)

API и бот работают так же, как раньше — клиенты (фронтенд, бот) не замечают смены хранилища.

**Сценарии приёмки:**

1. **Дано** бот запущен с SQLite, **Когда** владелец отправляет команду бронирования, **Тогда** ответ бота идентичен прежнему
2. **Дано** API работает с SQLite, **Когда** фронтенд запрашивает `/api/available-dates`, **Тогда** формат ответа не изменился

## Ключевые сущности

- **slots**: date_key (TEXT), hour (INTEGER), status (TEXT: free/booked), note (TEXT), client_name (TEXT), client_phone (TEXT)

## Edge Cases

- Что будет если файл БД повреждён при запуске? — сервер должен упасть с понятной ошибкой, а не молча работать с пустыми данными
- Что если volume не примонтирован? — файл БД создастся внутри контейнера и данные пропадут при пересоздании — добавить предупреждение в логах
