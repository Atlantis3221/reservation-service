# Фича: Nginx, домен, HTTPS

**Ветка:** `05-nginx-domain-https`
**Статус:** реализован

## Контекст

**Зачем:** демо должно быть доступно по нормальному URL с SSL-сертификатом. Сейчас бэкенд доступен только по IP:порт.

**Что есть сейчас:** бэкенд запускается в Docker на финском сервере, фронтенд деплоится на GitHub Pages. Нет единого домена, нет HTTPS для API.

## User Stories

### US-1 — Единый домен для всего сервиса (Приоритет: P1)

Пользователь заходит на `domain.ru` — видит фронтенд. Запросы к `/api/*` проксируются на бэкенд. Всё работает через HTTPS.

**Почему этот приоритет:** без домена и HTTPS нельзя показывать сервис клиентам — браузеры блокируют mixed content, URL с IP выглядит ненадёжно.

**Как проверить независимо:** открыть `https://domain.ru/slug-бани` в браузере, убедиться что сертификат валидный, фронтенд загружается, API отвечает.

**Сценарии приёмки:**

1. **Дано** сервер настроен, **Когда** пользователь открывает `https://domain.ru`, **Тогда** отображается фронтенд с валидным SSL
2. **Дано** фронтенд загружен, **Когда** он запрашивает `/api/business/test`, **Тогда** запрос проксируется на бэкенд и возвращает данные
3. **Дано** пользователь открывает `http://domain.ru`, **Тогда** он перенаправляется на `https://domain.ru`

---

### US-2 — Автообновление SSL (Приоритет: P1)

SSL-сертификат от Let's Encrypt автоматически обновляется, не требуя ручного вмешательства.

**Сценарии приёмки:**

1. **Дано** сертификат получен через Certbot, **Когда** срок действия подходит к концу, **Тогда** cron/timer обновляет сертификат автоматически

---

### US-3 — Health-check для мониторинга (Приоритет: P2)

Есть эндпоинт для проверки работоспособности сервиса.

**Сценарии приёмки:**

1. **Дано** сервис работает, **Когда** запрошен health-check, **Тогда** возвращается 200 OK

## Ключевые сущности

- **nginx config**: проксирование `/api/*` → backend:3000, отдача статики для `/`
- **certbot**: получение и обновление SSL от Let's Encrypt

## Edge Cases

- Что если certbot не может получить сертификат (DNS ещё не распространился)? — nginx должен работать на HTTP пока сертификат не получен
- Что если бэкенд упал? — nginx возвращает 502 с понятной страницей ошибки
