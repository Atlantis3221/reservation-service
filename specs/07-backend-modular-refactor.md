# Фича: Модульный рефакторинг бэкенда

**Ветка:** `07-backend-modular-refactor`
**Создана:** 2026-03-14
**Статус:** реализовано

## Контекст

**Зачем:** код бэкенда вырос органически — `bot.ts` (~420 строк) совмещает парсинг команд, дата-утилиты, бизнес-логику и форматирование ответов. `schedule.ts` перемешивает SQL-запросы с бизнес-правилами. Дата-хелперы дублируются между файлами. Это затрудняет навигацию, повторное использование и тестирование.

**Что есть сейчас:**

```
backend/src/
├── index.ts              — точка входа, Express, CORS
├── types.ts              — общие типы (SlotStatus, TimeSlot)
├── routes/
│   └── api.ts            — HTTP-эндпоинты (2 GET-маршрута)
└── services/
    ├── db.ts             — инициализация SQLite, миграции
    ├── bot.ts            — ~420 строк: парсинг + дата-хелперы + бизнес-логика + форматирование
    └── schedule.ts       — SQL-запросы + бизнес-правила в одних функциях
```

**Проблемы:**
1. `bot.ts` — god-файл: парсеры, утилиты дат, обработчики команд, форматирование ответов
2. `schedule.ts` — SQL перемешан с бизнес-логикой, нет data-слоя
3. `toDateKey` и другие дата-хелперы дублируются между `bot.ts` и `schedule.ts`
4. Неиспользуемые импорты в `api.ts` (`bookRange`, `setSlotStatus`)

## User Stories

### US-1 — Разделение bot.ts на модули (Приоритет: P1)

Бот-файл разбит на отдельные модули по ответственности: парсеры команд, дата-утилиты, форматирование ответов, обработчики команд.

**Почему этот приоритет:** `bot.ts` — самый крупный и запутанный файл. Любое изменение в боте требует ориентироваться в 420 строках смешанной логики.

**Как проверить независимо:** бот работает идентично — все команды дают те же ответы, что и до рефакторинга.

**Сценарии приёмки:**

1. **Дано** рефакторинг завершён, **Когда** владелец отправляет любую команду боту, **Тогда** ответ идентичен поведению до рефакторинга
2. **Дано** парсеры вынесены в отдельный модуль, **Когда** разработчик хочет изменить парсинг, **Тогда** он работает только с файлом парсеров, не трогая обработчики

---

### US-2 — Выделение data-слоя из schedule.ts (Приоритет: P1)

SQL-запросы вынесены в отдельный репозиторий/DAO. `schedule.ts` содержит только бизнес-логику и вызывает DAO для доступа к данным.

**Почему этот приоритет:** разделение позволяет менять способ хранения (например, переход на PostgreSQL) без переписывания бизнес-логики.

**Сценарии приёмки:**

1. **Дано** DAO выделен, **Когда** API запрашивает слоты, **Тогда** ответ идентичен прежнему
2. **Дано** DAO выделен, **Когда** разработчик хочет изменить SQL-запрос, **Тогда** он работает только с DAO-файлом

---

### US-3 — Общие дата-утилиты (Приоритет: P2)

Дублирующиеся дата-хелперы (`toDateKey`, `fmtDate`, `getMondayOfWeek` и др.) вынесены в один модуль `utils/date.ts` и используются из обоих мест.

**Сценарии приёмки:**

1. **Дано** утилиты вынесены, **Когда** используется `toDateKey` в боте и в schedule, **Тогда** оба импортируют из `utils/date.ts`
2. **Дано** утилиты вынесены, **Тогда** дублирующиеся определения удалены из `bot.ts` и `schedule.ts`

---

### US-4 — Чистка неиспользуемого кода (Приоритет: P2)

Удалены неиспользуемые импорты и мёртвый код.

**Сценарии приёмки:**

1. **Дано** рефакторинг завершён, **Тогда** в `api.ts` нет неиспользуемых импортов
2. **Дано** рефакторинг завершён, **Тогда** TypeScript компилируется без варнингов

## Ключевые сущности

Целевая структура после рефакторинга:

```
backend/src/
├── index.ts                  — точка входа (без изменений)
├── types.ts                  — общие типы (без изменений)
├── routes/
│   └── api.ts                — HTTP-эндпоинты (чистка импортов)
├── utils/
│   └── date.ts               — общие дата-утилиты (toDateKey, fmtDate, getMondayOfWeek, resolveDay и др.)
├── repositories/
│   └── slot.repository.ts    — SQL-запросы к таблице slots (чистый data access)
├── services/
│   ├── db.ts                 — инициализация SQLite (без изменений)
│   └── schedule.ts           — бизнес-логика расписания (вызывает repository)
└── bot/
    ├── index.ts              — инициализация Telegraf, регистрация хендлеров
    ├── handlers.ts           — обработчики команд (тонкие: парсинг → сервис → формат → ответ)
    ├── parsers.ts            — parseFlexibleSchedule, parseBookingCommand, parseBookingRange
    └── formatters.ts         — форматирование текстовых ответов (эмодзи, расписание, статистика)
```

## Edge Cases

- Что если при рефакторинге нарушается порядок инициализации (бот стартует до готовности БД)? — порядок в `index.ts` должен сохраниться: `initDb()` → `initBot()`
