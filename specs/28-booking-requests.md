# Фича: Заявки на бронирование от клиентов

**Ветка:** `28-booking-requests`
**Создана:** 2026-03-18
**Статус:** реализована

## Контекст

**Зачем:** Сейчас клиент, чтобы забронировать, должен перейти в мессенджер и написать владельцу вручную. Это создаёт трение: клиент уходит с сайта, не все пишут, часть клиентов теряется. Нужна возможность оставить заявку прямо на сайте — указать имя, телефон и опционально описание, а владелец получит уведомление и обработает заявку.

**Что есть сейчас:**
- На фронтенде (BusinessPage) при выборе даты показывается `booking-bar` с кнопками «Забронировать в Telegram/VK/MAX» — при клике копируется сообщение и открывается мессенджер.
- В админке есть вкладки «Чат» и «Календарь». Настройки бизнеса управляются через чат (бот-команды).
- Боты (TG и VK) поддерживают коллбэки `confirm_book` / `deny_book` для подтверждения/отмены броней.
- Бизнес имеет настройки: имя, slug, контактные ссылки. Нет настройки «форма заявки».

## User Stories

### US-1 — Настройка формы заявок владельцем (Приоритет: P1)

Владелец бизнеса включает/выключает приём заявок через сайт. Настройка доступна через бота (`/settings`) и через новую вкладку «Настройки» в админке. При включении — на фронтенде в модалке бронирования появляется форма заявки (в дополнение к существующим ссылкам на мессенджеры). При выключении — остаётся только текущее поведение со ссылками.

В админке создаётся блок «Настройки», куда выносятся все настройки бизнеса (вместо управления через чат).

**Почему этот приоритет:** Без настройки форма не появится, это основа фичи.

**Как проверить независимо:** В боте или админке включить форму заявок → на фронтенде должна появиться форма в модалке бронирования.

**Сценарии приёмки:**

1. **Дано** бизнес без формы заявок (по умолчанию выключена), **Когда** владелец включает форму заявок в настройках (бот или админка), **Тогда** на фронтенде при клике «Забронировать» в модалке видна форма + ссылки на мессенджеры.
2. **Дано** форма заявок включена, **Когда** владелец выключает её, **Тогда** на фронтенде форма пропадает, остаются только ссылки на мессенджеры (текущее поведение).
3. **Дано** бизнес с формой заявок выключенной, **Когда** клиент выбирает дату, **Тогда** поведение идентично текущему (кнопки мессенджеров).

### US-2 — Клиент отправляет заявку (Приоритет: P1)

Клиент на странице бизнеса выбирает дату, нажимает кнопку «Забронировать», видит модалку. В модалке — ссылки на мессенджеры (всегда) и форма заявки (если включена). Клиент выбирает время (как владелец — на таймлайне дня), указывает имя, телефон и опционально описание. После отправки видит «С вами свяжутся».

**Почему этот приоритет:** Ключевая ценность фичи — клиент оставляет заявку, не уходя с сайта.

**Как проверить независимо:** Открыть страницу бизнеса с включённой формой → заполнить → отправить → увидеть подтверждение.

**Сценарии приёмки:**

1. **Дано** бизнес с включённой формой, клиент на странице дня, **Когда** клиент нажимает «Забронировать», **Тогда** открывается модалка со ссылками на мессенджеры + форма (имя, телефон, время, описание).
2. **Дано** открыта модалка, **Когда** клиент заполняет имя, телефон, выбирает время и нажимает «Отправить», **Тогда** заявка сохраняется, клиент видит «С вами свяжутся».
3. **Дано** открыта модалка, **Когда** клиент не заполняет имя или телефон, **Тогда** кнопка «Отправить» неактивна или показывается ошибка валидации.
4. **Дано** клиент на странице конкретного дня, **Когда** открывает модалку, **Тогда** дата подставляется автоматически из выбранного дня.
5. **Дано** описание не заполнено, **Когда** клиент отправляет заявку, **Тогда** заявка создаётся успешно (описание опционально).

### US-3 — Уведомление владельца о новой заявке (Приоритет: P1)

При поступлении заявки владелец получает уведомление в Telegram-бот и VK-бот с данными заявки и кнопками «Подтвердить» / «Отклонить».

**Почему этот приоритет:** Без уведомления владелец не узнает о заявке, клиент останется без ответа.

**Как проверить независимо:** Отправить заявку на сайте → проверить что пришло сообщение в TG/VK бот.

**Сценарии приёмки:**

1. **Дано** заявка отправлена, **Когда** владелец подключён к TG-боту, **Тогда** он получает сообщение с именем клиента, телефоном, описанием, датой/временем и кнопками «Подтвердить» / «Отклонить».
2. **Дано** заявка отправлена, **Когда** владелец подключён к VK-боту, **Тогда** он получает аналогичное сообщение в VK.
3. **Дано** владелец нажимает «Подтвердить», **Тогда** заявка переходит в статус «подтверждена».
4. **Дано** владелец нажимает «Отклонить», **Тогда** заявка переходит в статус «отклонена».
5. **Дано** у бизнеса нет owner_chat_id (бот не привязан), **Когда** приходит заявка, **Тогда** заявка сохраняется, уведомление не отправляется. Заявку можно обработать в админке.

---

### US-4 — Вкладка заявок в админке (Приоритет: P2)

В админке появляется отдельная вкладка «Заявки», где видны все заявки с фильтрацией по статусу. Можно подтвердить, отклонить заявку или перенести запись на другое время (меняет preferred_date/time в заявке — реальный слот не создаётся до подтверждения).

**Почему этот приоритет:** Админка — дополнительный канал управления, боты уже покрывают основной сценарий.

**Как проверить независимо:** Открыть админку → перейти на вкладку «Заявки» → увидеть список заявок.

**Сценарии приёмки:**

1. **Дано** есть заявки со статусом «новая», **Когда** владелец открывает вкладку «Заявки», **Тогда** видны все заявки с именем клиента, телефоном, описанием, датой/временем и статусом.
2. **Дано** заявка со статусом «новая», **Когда** владелец нажимает «Подтвердить», **Тогда** заявка переходит в «подтверждена».
3. **Дано** заявка со статусом «новая», **Когда** владелец нажимает «Отклонить», **Тогда** заявка переходит в «отклонена».
4. **Дано** заявка со статусом «новая» или «подтверждена», **Когда** владелец нажимает «Перенести» и выбирает другую дату/время, **Тогда** в заявке обновляется дата/время. Клиент не уведомляется.

---

### US-5 — Страница настроек в админке (Приоритет: P2)

В админке создаётся вкладка «Настройки» с блоком управления всеми настройками бизнеса. Настройки выносятся из чата сюда. Включает: вкл/выкл формы заявок, контактные ссылки, имя бизнеса, slug.

**Почему этот приоритет:** Централизация настроек улучшает UX админки.

**Как проверить независимо:** Открыть вкладку «Настройки» → увидеть и изменить настройки бизнеса.

**Сценарии приёмки:**

1. **Дано** владелец открыл админку, **Когда** переходит на вкладку «Настройки», **Тогда** видит все настройки бизнеса: имя, slug, контактные ссылки, переключатель формы заявок.
2. **Дано** владелец на вкладке «Настройки», **Когда** включает/выключает форму заявок, **Тогда** изменение сохраняется и сразу отражается на фронтенде.

## Ключевые сущности

- **BookingRequest (заявка)**: id, business_id, client_name, client_phone, description (опционально), preferred_date, preferred_time, status (pending/approved/rejected), created_at, updated_at
- **Business.booking_requests_enabled**: булевый флаг — принимает ли бизнес заявки через сайт (по умолчанию false)

## Edge Cases

- Клиент отправил заявку, а у бизнеса нет owner_chat_id (бот не привязан) → Заявка сохранится, уведомление не уйдёт. Заявку можно обработать в админке.
- Клиент отправил несколько заявок подряд → Пока без ограничений (rate limiting).
- Невалидный телефон → Базовая валидация формата на фронтенде.
- Модалка «Забронировать» когда форма выключена → Показываются только ссылки на мессенджеры (текущее поведение).
- Модалка «Забронировать» когда нет ни ссылок, ни формы → Сообщение «Свяжитесь с владельцем для бронирования» (текущее поведение).
