# Фича: Защита данных от потери при деплое

**Ветка:** `30-data-safety`
**Создана:** 2026-03-19
**Статус:** реализовано

## Контекст

**Зачем:** При деплое на продакшн SQLite-база данных может быть пересоздана с нуля. Это приводит к потере `admin_users` (и связки с бизнесами через `owner_chat_id`). Бизнесы при этом пересоздаются Telegram-ботом, но admin-аккаунт теряет привязку. Пользователь может зарегистрироваться повторно на тот же email — и потерять доступ к своим заведениям.

**Что есть сейчас:**
- SQLite-файл хранится в `./data/reservations.db` через Docker volume bind mount `./data:/app/data`
- Миграции через `CREATE TABLE IF NOT EXISTS` — не обновляют существующие таблицы и не отслеживают версию схемы
- При деплое (`deploy.yml`) rsync исключает `data/`, но нет бэкапа перед перезапуском контейнера
- Нет никакой защиты от потери volume (первый деплой, смена сервера, ручная очистка)
- `register()` проверяет дубликат email в коде + UNIQUE constraint в схеме, но если БД пуста — проверки проходят

## Проблемы (корневые причины)

1. **Нет бэкапов** — если данные пропали, восстановить невозможно
2. **Миграции не версионированы** — `CREATE TABLE IF NOT EXISTS` не обновляет существующие таблицы (нельзя добавить UNIQUE к уже существующей таблице)
3. **Нет валидации целостности при старте** — приложение не замечает, что БД пустая/повреждённая
4. **Потеря связки admin↔business** — если admin_user пересоздан, owner_chat_id теряется

## User Stories

### US-1 — Автоматический бэкап БД при деплое (Приоритет: P1)

При каждом деплое, перед перезапуском контейнеров, создаётся бэкап SQLite-файла на сервере.

**Почему этот приоритет:** Без бэкапа данные невосстановимы. Это минимально необходимая защита.

**Как проверить независимо:** Запустить деплой, убедиться что перед `docker-compose up` создаётся копия `data/reservations.db` с timestamp.

**Сценарии приёмки:**

1. **Дано** продакшн-сервер с существующей БД, **Когда** запускается деплой, **Тогда** перед перезапуском контейнеров создаётся файл `data/backups/reservations-YYYY-MM-DD-HHmmss.db`
2. **Дано** директория `data/backups/` с > 10 бэкапами, **Когда** создаётся новый бэкап, **Тогда** самые старые бэкапы удаляются (остаётся не более 10)

---

### US-2 — Бэкап БД при старте приложения (Приоритет: P1)

Приложение при запуске создаёт бэкап существующей БД (если она не пустая).

**Почему этот приоритет:** Контейнер может перезапуститься не только через CI/CD, но и через Docker restart policy.

**Как проверить независимо:** Перезапустить контейнер, увидеть в логах строку `[db] Backup created: ...`

**Сценарии приёмки:**

1. **Дано** приложение с существующей БД, **Когда** приложение стартует, **Тогда** создаётся файл `data/backups/reservations-YYYY-MM-DD-HHmmss.db` и выводится лог
2. **Дано** пустая БД (0 таблиц или 0 записей), **Когда** приложение стартует, **Тогда** бэкап НЕ создаётся (нечего бэкапить)
3. **Дано** директория `data/backups/` с > 10 бэкапами, **Когда** создаётся новый бэкап, **Тогда** остаётся не более 10 файлов

---

### US-3 — Версионированные миграции (Приоритет: P2)

Вместо `CREATE TABLE IF NOT EXISTS` — пронумерованные миграции с отслеживанием текущей версии в таблице `_migrations`.

**Почему этот приоритет:** Позволяет безопасно изменять схему (добавлять constraints, переименовывать колонки) без потери данных.

**Как проверить независимо:** Добавить новую миграцию, запустить приложение, убедиться что она применилась и версия обновилась.

**Сценарии приёмки:**

1. **Дано** свежая БД без таблиц, **Когда** приложение стартует, **Тогда** все миграции применяются по порядку, таблица `_migrations` содержит запись с текущей версией
2. **Дано** БД с версией 5, **Когда** приложение стартует с миграциями до версии 7, **Тогда** применяются только миграции 6 и 7
3. **Дано** миграция с ошибкой, **Когда** она применяется, **Тогда** все изменения откатываются (транзакция), приложение логирует ошибку

---

### US-4 — Предупреждение о пустой/подозрительной БД (Приоритет: P2)

При старте приложение проверяет состояние БД. Если БД выглядит пустой или только что пересозданной (0 бизнесов, 0 admin_users) — пишет WARN в лог и отправляет уведомление в мониторинг-бот.

**Сценарии приёмки:**

1. **Дано** приложение в production, **Когда** БД содержит 0 строк в `businesses` или `admin_users`, **Тогда** выводится WARNING в лог и отправляется сообщение в мониторинг-бот

---

## Ключевые сущности

- **Backup** — копия SQLite-файла с timestamp, хранится в `data/backups/`
- **_migrations** — таблица с одной колонкой `version INTEGER`, отслеживает текущую версию схемы
- **Migration** — функция `(db) => void`, применяемая в транзакции, с номером версии

## Edge Cases

- Что если `data/backups/` не существует? → Создать автоматически
- Что если диск заполнен и бэкап не создаётся? → Логировать ошибку, не падать
- Что если миграция падает посередине? → Откатить транзакцию, приложение стартует со старой версией, логирует ошибку
- Что если БД повреждена (corrupt)? → SQLite PRAGMA integrity_check при старте, WARNING в лог

## Критерии приёмки

- [ ] При деплое через CI/CD создаётся бэкап `data/backups/reservations-*.db` перед перезапуском контейнера
- [ ] При старте приложения создаётся бэкап (если БД не пустая), хранится максимум 10 последних
- [ ] Миграции пронумерованы, применяются последовательно, версия хранится в `_migrations`
- [ ] Существующая миграция из `migrate()` разбита на версионированные шаги
- [ ] При пустой БД в production — WARNING в лог
- [ ] Все миграции оборачиваются в транзакции
- [ ] Бэкап не блокирует запуск приложения при ошибке (fail-safe)
