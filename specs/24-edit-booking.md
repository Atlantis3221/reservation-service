# Фича: Редактирование записи

**Ветка:** `24-edit-booking`
**Создана:** 2026-03-17
**Статус:** реализовано

## Контекст

**Зачем:** Владелец заведения не может изменить данные уже созданной записи — приходится отменять старую и создавать новую. Это неудобно, особенно когда клиент просит перенести время или нужно исправить опечатку в имени/телефоне.

**Что есть сейчас:** В админ-панели (визуальный календарь) при клике на запись открывается bottom sheet с деталями в режиме просмотра (время, имя клиента, телефон, заметка). Единственное доступное действие — отмена записи. Редактирование отсутствует.

## User Stories

### US-1 — Редактирование данных записи из визуального календаря (Приоритет: P1)

Владелец заведения кликает на существующую запись (статус `booked`) в визуальном календаре, видит её данные и может отредактировать любое поле: время начала/конца, имя клиента, телефон, заметку.

**Почему этот приоритет:** Базовый сценарий — клиент просит перенести время или владелец хочет дополнить информацию. Без этого приходится отменять и создавать запись заново.

**Как проверить независимо:** Открыть календарь → кликнуть на запись → изменить поле → сохранить → убедиться что данные обновились.

**Сценарии приёмки:**

1. **Дано** запись со статусом `booked`, **Когда** владелец кликает на неё, **Тогда** открывается bottom sheet с данными записи и кнопкой «Редактировать»
2. **Дано** открыт bottom sheet записи, **Когда** владелец нажимает «Редактировать», **Тогда** поля становятся редактируемыми (время, имя, телефон, заметка)
3. **Дано** владелец изменил имя клиента и нажал «Сохранить», **Когда** запрос успешен, **Тогда** sheet закрывается, календарь обновляется, на записи отображается новое имя
4. **Дано** владелец изменил время записи, **Когда** новое время пересекается с другой записью, **Тогда** показывается предупреждение о конфликте с возможностью подтвердить или отменить
5. **Дано** владелец не менял ни одного поля, **Когда** нажимает «Сохранить», **Тогда** sheet просто закрывается без запроса к серверу

---

### US-2 — Валидация при редактировании (Приоритет: P1)

При сохранении изменений система проверяет корректность данных.

**Почему этот приоритет:** Без валидации можно сохранить некорректные данные.

**Сценарии приёмки:**

1. **Дано** владелец очистил поле «Имя клиента», **Когда** нажимает «Сохранить», **Тогда** показывается ошибка «Укажите имя клиента»
2. **Дано** владелец указал время начала позже времени конца (например 22:00–02:00), **Когда** нажимает «Сохранить», **Тогда** запись сохраняется как бронирование через полночь

## Ключевые сущности

- **Slot (запись)**: существующая сущность в таблице `slots`, поля: `id`, `business_id`, `date_key`, `start_time`, `end_time`, `status`, `note`, `client_name`, `client_phone`. Редактирование — это UPDATE существующей строки.

### US-3 — Перенос записи на другую дату (Приоритет: P1)

Владелец может изменить дату записи — перенести на другой день. При смене даты проверяются конфликты на новую дату.

**Почему этот приоритет:** Клиент часто просит перенести визит на другой день, не только на другое время.

**Сценарии приёмки:**

1. **Дано** запись на 20 марта 14:00–16:00, **Когда** владелец меняет дату на 22 марта и нажимает «Сохранить», **Тогда** запись перемещается на 22 марта 14:00–16:00
2. **Дано** на 22 марта уже есть запись 14:00–16:00, **Когда** владелец переносит на 22 марта 14:00–16:00, **Тогда** показывается предупреждение о конфликте

## Edge Cases

- Одновременное редактирование одной записи двумя админами — last-write-wins, без optimistic locking
- Записи через полночь (`end_time < start_time`) — при редактировании времени логика сохраняется
- Перенос записи на другой день — `date_key` обновляется вместе с остальными полями, проверяются конфликты на новую дату

## Критерии приёмки

- [ ] В bottom sheet записи со статусом `booked` есть кнопка «Редактировать»
- [ ] По нажатию «Редактировать» поля (дата, время начала, время конца, имя клиента, телефон, заметка) становятся редактируемыми
- [ ] При сохранении без изменений sheet закрывается без запроса к серверу
- [ ] При изменении любого поля и нажатии «Сохранить» данные обновляются на сервере и календарь перерисовывается
- [ ] При конфликте по времени показывается предупреждение с возможностью подтвердить
- [ ] Валидация: имя клиента обязательно
- [ ] API `PUT /admin/calendar/booking/:id` принимает обновлённые поля и возвращает результат
- [ ] Проверка прав доступа: только владелец заведения может редактировать свои записи
