# Фича: Произвольное время начала и конца бронирования

**Ветка:** `13-flexible-time`
**Создана:** 2026-03-16
**Статус:** реализовано

## Контекст

**Зачем:** Сейчас система позволяет создавать бронирования только на целые часы (например, 14:00–16:00). Владельцу бизнеса нужна возможность бронировать на произвольное время — например, с 10:14 до 12:30, чтобы точнее отражать реальное расписание.

**Что есть сейчас:**
- В БД слоты хранятся как `date_key` (YYYY-MM-DD) + `hour` (integer 0–23) — только целые часы
- `TimeSlot.datetime` всегда формируется как `YYYY-MM-DDTHH:00:00`, `duration` = 1 (час)
- Telegram-бот при парсинге явно отвергает ненулевые минуты (`minutes !== 0 → return null`)
- Фронтенд показывает часовую сетку (10:00–22:00), клик по часу — нет выбора минут
- Длительность на фронте выбирается как 1/2/3/4 часа

## User Stories

### US-1 — Бронирование через бот с произвольным временем (Приоритет: P1)

Владелец бизнеса создаёт бронь через Telegram-бот, указывая точное время начала и конца с минутами.

**Почему этот приоритет:** Основной способ создания броней — через бот. Без поддержки минут фича не работает.

**Как проверить независимо:** Отправить боту команду типа «бронь сегодня с 10:14 до 12:30 Иванов» и убедиться, что бронь создана с правильным временем.

**Сценарии приёмки:**

1. **Дано** бот настроен, **Когда** отправляю «бронь сегодня с 10:15 до 12:30 Иванов», **Тогда** создаётся бронь с 10:15 до 12:30
2. **Дано** бот настроен, **Когда** отправляю «бронь сегодня с 14 до 16 Петров», **Тогда** создаётся бронь с 14:00 до 16:00 (обратная совместимость)
3. **Дано** бот настроен, **Когда** отправляю «бронь сегодня на 14:30 на 2 часа Сидоров», **Тогда** создаётся бронь с 14:30 до 16:30

---

### US-2 — Отображение произвольных броней на фронтенде (Приоритет: P1)

Пользователь видит на фронтенде бронирования с точным временем, а не округлённым до часа.

**Почему этот приоритет:** Фронт — основной интерфейс просмотра расписания для клиентов.

**Как проверить независимо:** Создать через бот бронь с минутами, открыть фронтенд, убедиться что время отображается корректно (например, «10:15 – 12:30»).

**Сценарии приёмки:**

1. **Дано** есть бронь 10:15–12:30, **Когда** открываю дневной вид, **Тогда** бронь отображается с правильным временем и визуально занимает корректный диапазон
2. **Дано** есть бронь 14:00–16:00 (целые часы), **Когда** открываю дневной вид, **Тогда** бронь отображается так же, как раньше

---

### US-3 — Выбор произвольного времени на фронтенде (Приоритет: P2)

Пользователь при создании заявки на бронирование на фронтенде может выбрать произвольное время начала и конца.

**Почему этот приоритет:** Фронтенд сейчас только формирует сообщение для Telegram, но UX должен поддерживать минуты.

**Как проверить независимо:** Открыть фронтенд, выбрать дату, задать произвольное время начала и конца — в сформированном сообщении видно точное время.

**Сценарии приёмки:**

1. **Дано** выбрана дата, **Когда** указываю время начала 10:15 и конца 12:30, **Тогда** формируется сообщение с «с 10:15 до 12:30»
2. **Дано** выбрана дата, **Когда** указываю конец раньше начала, **Тогда** показывается ошибка валидации

## Ключевые сущности

- **Slot (обновлённый)**: вместо `hour` (integer) → `start_time` (TEXT, "HH:MM") + `end_time` (TEXT, "HH:MM"). Каждый слот — одно бронирование с произвольным диапазоном времени, а не часовая ячейка.
- **TimeSlot (TypeScript)**: `datetime` заменяется на `startDatetime` (ISO, `YYYY-MM-DDTHH:MM:00`) + `endDatetime` (ISO), `duration` убирается или вычисляется.

## Edge Cases

- **Пересечения бронирований**: в этой фиче проверка пересечений не реализуется — владелец сам следит. Проверка пересечений — отдельная фича.
- **Миграция данных**: существующие слоты (hour=14) → start_time="14:00", end_time="15:00"
- **Минимальная длительность**: ограничений нет, любая длительность допустима
- **Шаг минут на фронтенде**: 5 минут (10:00, 10:05, 10:10…)
- **Бронь через полночь**: допускается (например, 23:00–01:00). end_time < start_time означает переход через полночь.

## Критерии приёмки

- [ ] Бот принимает команду «бронь сегодня с 10:15 до 12:30 Иванов» и создаёт бронь с точным временем
- [ ] Бот принимает команду «бронь сегодня на 14:30 на 2 часа Сидоров» и создаёт бронь 14:30–16:30
- [ ] Бот продолжает принимать целые часы «бронь сегодня с 14 до 16 Петров» (обратная совместимость)
- [ ] В БД слоты хранятся с `start_time` и `end_time` в формате "HH:MM"
- [ ] Миграция конвертирует существующие данные (hour → start_time/end_time)
- [ ] API `GET /api/business/:slug/day-slots` возвращает `startDatetime` и `endDatetime`
- [ ] Фронтенд отображает брони с точным временем (визуально корректная позиция и высота блока)
- [ ] На фронтенде время выбирается с шагом 5 минут
- [ ] Фронтенд позволяет выбрать время начала и время конца (вместо «длительность в часах»)
- [ ] Бронь через полночь (23:00–01:00) корректно создаётся и отображается
- [ ] Валидация: время конца не может быть равно времени начала
